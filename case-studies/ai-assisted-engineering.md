# AI-assisted Engineering & Knowledge Work

## Context

AI tools became part of my engineering workflow gradually: first as a learning and coding aid during my Java/Spring retraining, then as daily support for documentation, debugging, infrastructure automation, review, and since early 2026 also agentic coding workflows on repository-level tasks.

I use AI as a multiplier, not as a replacement for engineering ownership. Architecture, security, validation, production decisions, and final review stay with me.

## Workflows

- Code exploration, implementation support, refactoring options, and test ideas.
- Code review, debugging hypotheses, log interpretation, and edge-case discovery.
- Documentation drafting, runbook structure, migration summaries, and portfolio synthesis.
- Infrastructure and operations support around Ansible, Docker, monitoring, and incident-oriented workflows.
- Personal knowledge management and assistant workflows for notes, tasks, ideas, and decision support.

## Tools

```text
OpenAI / ChatGPT / Codex · Claude · Gemini · GitHub Copilot
agentic coding workflows · AI-supported documentation
personal knowledge management / personal assistant tooling
```

## Principles

- Generated code is treated like code from another engineer: read it, test it, adapt it to local patterns.
- No secrets, customer data, internal URLs, ticket exports, or private raw material go into external tools.
- Different models are useful for different tasks, so I compare outputs rather than trusting a single answer blindly.
- AI is strongest when paired with clear constraints, repository context, tests, and human judgment.

## Outcome

- Faster exploration of unfamiliar code and operational problems.
- Better documentation and handover material.
- More systematic review of implementation options, failure modes, and edge cases.
- A practical, production-aware AI workflow that fits backend, platform, DevOps, and leadership work.
